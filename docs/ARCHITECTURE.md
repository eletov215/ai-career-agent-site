# AI Career Agent — архитектура проекта

> Статус документа: рабочая версия  
> Последнее обновление: 20 июля 2026 года

## 1. Назначение проекта

AI Career Agent — веб-приложение для поиска и анализа вакансий из нескольких источников.

Основная идея:

1. Пользователь задаёт параметры поиска.
2. Система получает вакансии из доступных источников.
3. Вакансии приводятся к единому формату.
4. Пользователь самостоятельно выбирает подходящие вакансии.
5. В дальнейшем система может помогать с анализом вакансии и подготовкой ответов, но не должна отправлять отклики без явного действия пользователя.

## 3. Запуск приложения

Render запускает приложение командой:

```bash
gunicorn app:app
```

Поэтому главным исполняемым файлом является:

```text
app.py
```

Файлы наподобие `app_fixed.py` не должны использоваться.

## 5. Основные компоненты

### 5.1 `app.py`

Отвечает за:

- создание Flask-приложения;
- маршруты сайта;
- обработку форм;
- сессии пользователя;
- OAuth HeadHunter;
- запуск поиска вакансий;
- передачу результатов в HTML-шаблоны;
- подключение провайдеров вакансий.

### 5.2 `services/hh_provider.py`

Отвечает за:

- запрос вакансий к HeadHunter;
- передачу параметров поиска;
- использование OAuth-токена при наличии;
- повторный публичный запрос при `401` или `403`;
- преобразование ответа HH в единый формат;
- логирование ошибок.

Текущая проблема:

```text
GET https://api.hh.ru/vacancies
HTTP 403 Forbidden
```

Запрос с токеном и повторный запрос без токена возвращают одинаковый результат.

В ответе присутствует:

```text
Server: ddos-guard
```

Пример `request_id`:

```text
17845451484092e03e804cc4bfe38025
```

Вероятная причина — ограничение или блокировка на стороне HH/DDOS-GUARD либо исходящего IP Render. Это пока не подтверждено официально.

### 5.3 Trudvsem

Интеграция с «Работа России» используется как независимый источник вакансий.

Необходимо поддерживать:

- получение вакансий;
- синхронизацию;
- сохранение статуса последнего запуска;
- обработку ошибок;
- отображение результатов в общем поиске.

### 5.4 Шаблоны

HTML-шаблоны отвечают только за представление данных.

В них не следует помещать:

- сетевые запросы;
- бизнес-логику;
- обработку OAuth;
- работу с токенами.

### 5.5 Статические файлы

Папка `static/` содержит:

- CSS;
- JavaScript;
- изображения интерфейса.

## 6. Единый формат вакансии

Все источники должны приводить вакансии к общей структуре:

```python
{
    "id": "...",
    "source": "hh",
    "title": "...",
    "company": "...",
    "salary_from": None,
    "salary_to": None,
    "currency": None,
    "area": "...",
    "remote": False,
    "url": "...",
    "published_at": "...",
    "description": "...",
}
```

Дополнительные поля допустимы, но базовые поля должны сохраняться.

## 7. Поток поиска

```text
Форма поиска
    ↓
Маршрут Flask
    ↓
Выбранные провайдеры
    ↓
Запросы к API
    ↓
Нормализация
    ↓
Объединение результатов
    ↓
Шаблон vacancies_unified.html
```

Ошибка одного источника не должна ломать результаты других источников.

## 8. OAuth HeadHunter

Рабочий поток:

```text
/oauth/hh/login
    ↓
Переход на HH
    ↓
/oauth/hh/callback
    ↓
Проверка state
    ↓
Получение access_token
    ↓
Сохранение подключения
```

Требования безопасности:

- проверять `state`;
- не выводить `access_token` в логи;
- не выводить `client_secret`;
- не показывать OAuth `code`;
- хранить секреты только в переменных окружения Render;
- не хранить секреты в GitHub.

## 9. Ограничения API HeadHunter

Поддержка HH сообщила:

- API соискателя больше не поддерживается;
- токен соискателя недоступен;
- официальная отправка откликов через API невозможна;
- доступен API работодателя;
- поиск вакансий рассматривается отдельно через открытые методы.

Поэтому архитектура проекта не должна зависеть от автоматической отправки откликов через API HH.

## 10. Обработка ошибок

Каждый провайдер должен возвращать контролируемый результат:

```python
SearchResult(
    items=[],
    page=0,
    pages=0,
    error="Описание ошибки",
)
```

Нельзя допускать падение всей страницы из-за ошибки одного API.

## 11. Логирование

В логах допустимо хранить:

- URL без секретов;
- HTTP-статус;
- имя провайдера;
- время выполнения;
- `request_id`;
- тип ошибки.

В логах запрещено хранить:

- access token;
- refresh token;
- client secret;
- OAuth code;
- пароли;
- содержимое пользовательских сессий.

## 13. Архитектурные правила

1. GitHub — главный источник актуального кода.
2. Изменяется только актуальный рабочий файл.
3. Новый файл не создаётся как замена старому без необходимости.
4. Провайдеры вакансий должны быть изолированы.
5. Ошибка одного провайдера не останавливает весь поиск.
6. Все секреты находятся в переменных окружения.
7. Каждое изменение фиксируется в `CHANGELOG.md`.
8. Каждая завершённая версия получает номер.
